# 主流大模型“知识图谱”式总结（按家族/能力维度）

本文档目标：帮你在脑海里建立一个“模型世界地图”。不尝试穷举每个具体型号，而是用**家族（供应商/开源系）+ 能力维度（推理/代码/多模态/成本/部署）**来组织理解，便于后续你看到任何新模型都能快速归类。

> 约束说明：模型更新非常快，“全量列举所有型号”不可行且很快过时；因此本文以“分类法 + 代表家族特征 + 选型规则”为主。

## 0. 本轮更新（全景骨架）

本轮目标是把“知识图谱”升级成可用的两张表：

- **能力矩阵（家族级）**：你看到新模型，先按家族归类，再按能力标签做对比。
- **选型指南（任务导向）**：写作/代码/推理/多模态分别怎么选“主力模型 + 配套模型”。

## 1. 怎么看模型：先看 6 个维度

1. **定位**：通用对话 / 推理强化 / 代码强化 / 多模态强化 / 审核与安全 / 小模型端侧
2. **能力**：工具调用、结构化输出（JSON/schema）、多轮一致性、长上下文、流式输出
3. **质量**：推理正确率、代码可运行率、写作风格、遵循指令能力
4. **成本与时延**：单次调用成本、吞吐、首 token 延迟、是否支持缓存
5. **可控性**：是否容易“按格式输出”、是否容易跑偏、是否更容易被提示词操控/注入
6. **部署形态**：云 API / 私有化 / 本地（Ollama/vLLM 等）/ 端侧（手机/嵌入式）

## 2. 市面模型的大类（你可以当成“地图层”）

### 2.1 闭源云模型（SaaS/API）

特点：

- 迭代快、综合能力强（通常也是新能力的首发地：更强多模态、更强工具调用、更强结构化输出）
- 依赖网络与供应商；成本随用量增长；可控性与合规要做额外工程

常见供应商类型（不穷举）：

- 通用旗舰/全能：OpenAI、Google、Anthropic 等
- 推理强化：常以 “reasoning/思维链/推理版” 系列存在（不同厂商命名不同）
- 国内云模型：阿里、字节、百度、腾讯等（通常中文与成本/吞吐更有优势，生态/工具对接各有侧重）

### 2.2 开源模型（可自部署）

特点：

- 成本与隐私可控（尤其在大流量或敏感数据场景），可做深度定制（微调、蒸馏、量化）
- 综合能力通常落后于最强闭源旗舰，但在“足够好 + 更便宜 + 可控”场景非常划算

开源“家族”更重要（型号只是家族的某个版本）：

- Llama 系（Meta）
- Qwen 系（阿里）
- Mistral 系（Mistral）
- Gemma 系（Google）
- DeepSeek 开源系
- Phi 系（Microsoft）
- Granite 系（IBM）
- Yi 等其它区域性/社区模型

### 2.3 专用模型（能力单点极强）

特点：不是“聊天最聪明”，但在单点能力上更可靠/更便宜。

- **Embedding**：向量检索/RAG
- **Moderation/Safety**：内容审核（例如 Llama Guard 这类）
- **ASR/TTS/Audio**：语音输入输出
- **OCR/Document**：文档理解与结构抽取

## 3. 按“任务”理解模型：你关心的四类

### 3.1 写作/内容模型：看“风格一致性 + 发散可控”

关键点：

- 是否能稳定遵循你的写作模板（章节骨架、黄金三章结构等）
- 是否擅长中文语感/网文节奏（这点往往与语料与对齐策略有关）
- 温度（temperature）调高是否“更有灵感”还是“更跑偏”
- 长文是否容易丢设定（需要：长上下文、记忆、压缩、结构化设定）

工程建议：

- “设定/人物卡”尽量用结构化输出落盘（JSON/schema），再把设定作为事实注入
- 长对话必上 history compression 或“阶段摘要”

### 3.2 代码/工程模型：看“可运行率 + 工具调用 + 结构化输出”

关键点：

- 是否能稳定输出可编译/可运行代码（尤其强类型语言）
- 是否能遵循项目约束（不引入不存在的依赖、按既有架构改）
- 是否擅长工具调用：检索/读文件/跑测试/静态分析（有些模型“会用工具”，有些只会聊天）
- 是否能输出结构化结果：例如输出 JSON patch、变更清单、API 结构等

工程建议：

- 代码生成任务尽量配合工具链：静态检查、单测、lint、format
- 多步骤任务更适合“planner/agentic loop”，而不是一次性长提示词

### 3.3 推理/数学模型：看“稳定正确 + 可验证”

关键点：

- 是否擅长多步推理、反事实、严谨约束
- 是否容易编造（hallucination）而不自知

工程建议：

- 尽量把推理拆成可验证步骤：中间产物结构化、对关键结论做二次校验
- 对“必须正确”的场景，考虑引入规则/约束求解/外部验证工具

### 3.4 多模态模型：看“输入模态 + 任务类型”

常见能力维度：

- 图片理解（caption、OCR、UI理解、图表解释）
- 视频理解（更依赖供应商支持）
- 音频理解/生成（ASR/TTS/对话）

工程建议：

- 多模态往往更吃 token/成本，尽量先做裁剪与结构化（例如先 OCR，再走文本模型）

## 4. “能力标签”速查（看到新模型怎么归类）

你可以把任何模型贴上这些标签：

- **Tools**：能否产出 tool call（以及 toolChoice 能否控制“必须/禁止/强制某工具”）
- **Schema/Structured Output**：能否可靠输出 JSON/schema（Basic vs Standard）
- **Multiple Choices**：能否一次给多候选（用于多方案）
- **Context Length**：长上下文上限（决定你能塞多少历史/设定/代码）
- **Streaming**：能否流式输出（体验/时延）
- **Moderation**：是否有审核模型/能力可用（输入输出安全）
- **Prompt Caching**：是否支持缓存（重复改稿/重复问答降成本）

## 5. 开源 vs 闭源：快速决策规则

选择闭源优先：

- 你要最强综合能力、最强多模态、最强工具调用与生态
- 你能接受数据上云与按量计费

选择开源/自部署优先：

- 你有隐私/合规/内网需求
- 你有稳定的大流量，算得过账（GPU/推理服务）
- 你需要深度定制（微调、领域适配、专用格式输出）

## 6. 给“写作助手 + 工具链”的选型建议（可直接用）

如果你的目标是“写作助手（长文）+ 工具（灵感/人物卡/资料/审稿）”：

1. **核心模型（主写作）**：优先选择长上下文 + 写作风格稳的通用模型（你会反复改稿与续写）
2. **结构化模型（设定落盘）**：优先选择结构化输出稳定（JSON/schema/严格模式）的模型（人物卡/世界观/分章结构要落盘）
3. **工具模型（资料与事实）**：优先选择工具调用稳定的模型（检索/数据库/引用输出）
4. **审核模型（建议配套）**：输入/输出/工具内容三段式审核，避免设定/素材里出现风险内容
5. **成本模型（批量活）**：用于批量润色、换风格、抽取要点、分镜等“量大但不要求最强推理”的任务

按任务给一个“配方”：

- **长文写作主线**：主写作模型（风格/中文） + history compression（阶段摘要） + 设定结构化落盘
- **资料型写作**：工具模型（能稳用工具） + 结构化模型（能稳出引用/表格/JSON） + 主写作模型（最后润色成文风）
- **多模态辅助写作**：多模态模型先 OCR/理解图片/表格 -> 再把结果喂给主写作模型
